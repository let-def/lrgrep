\section{Theoretical fundation and concepts behind LRgrep}

\begin{frame}{Characterizing errors (1/3): langage complement}
  A grammar $G$ denotes a language $L$, a subset of the words on some alphabet $\Sigma$ (also known as terminals).
  $$
  [\![G\,]\!] = L \subset \Sigma^*
  $$

  \pause
  As a first approximation, an invalid input is any word that does not belong to the language:
  $$
  \Sigma^* / L
  $$

  \pause
  The grammar also gives structure to the words: the grammatical derivations.
  \\
  Can we give some structure to the errors?
\end{frame}

\begin{frame}{Characterizing errors (2/3): the error front}
  An LR parser consumes the input from left to right, and fail as early as possible (as soon as the input is not a prefix of any word of the language).

  % Attention aux conflits cependant !
  % C'est grâce à ça qu'on peut raporter au moins la position de l'erreur.

  \pause
  Lets considers this miminal set of failing prefixes:

  $$
  \mathcal E_L = \left\{ (u, t) \in \Sigma^*\times\Sigma\ \bigg|\ \bigwedge
  \begin{array}{l}
      \exists v \in \Sigma^* \ u . v \in L \\
      \nexists v' \in \Sigma^* \ u . t . v' \in L
  \end{array}
  \right\}
  $$

  In this set of pairs $(u,t)$, $u$ is a valid prefix and $t$ is a terminal that will cause the analysis to fail.
\end{frame}

\begin{frame}{Characterizing errors (3/3): regularisation}
  Since the $u$-prefixes are valid, we can feed them to a parser and get a {\em stack} (technically a {\em right sentential form}).

  Let's call $im$ the function associating a valid prefix to a stack. We define:

  $$
  \mathcal E_G = \left\{\ (im(u), t)\ \mid\ (u, t) \in \mathcal E_L\ \right\}
  $$

  \pause
  \begin{block}{Theorem}
    The set of all viable prefixes of the right sentential forms of a grammar is a regular language.
  \end{block}

  \pause

  Therefore $\mathcal E_G$ is a regular set!

  \textbf{Good news:} many properties are decidable on regular sets, we have
  many tools to work with them (such as regular expressions).

\end{frame}

\begin{frame}{Intuition on ``Shift-Reduce'' parsers}

  {\em Bottom-up} analysis:
  \begin{itemize}
    \item the input is consumed left to right
    \item the derivation is built starting from the leaves,
          that are progressibely rooted.
    \item a stack is used to save the intermediate (sub-)derivations
  \end{itemize}

  \pause

  Example with a minimalist grammar:
  $$
  \begin{array}{rl}
    exp ::= & num \\
          | & exp\,+ exp \text{, right associative}\\
  \end{array}
  $$

  Let's analyse the expresssion $1+2+3$.
\end{frame}

\begin{frame}[t]{Regular expressions for LR stacks}
  $$
  \begin{array}{rlll}
    e ::=& X      & & \text{Symbol (terminal or non-terminal)}\\
    | & e_1 e_2 & & \text{Concatenation}\\
    | & e_1 | e_2 & & \text{Disjunction}\\
    | & e^*       & & \text{Kleene star}\\
    \only<1-2>{| & \ldots &&}
    \only<3->{| & [e\,]       & & \text{{\em Handles}}}
  \end{array}
  $$
  \pause
  \begin{minipage}{10cm}
    \vspace{1cm}
  \only<1-2>{
    \uncover<2>{
    Enough to identify specific stacks, too fine-grained in practice:
    no way to talk about reductions.
    }
  }
  \end{minipage}
  \begin{minipage}{10cm}
    \vspace{1cm}
  \only<3->{
    \uncover<4->{
      We add a new construct $[e\,]$ to denote a set of {\em handles}
      that can reduce, in one or more steps, to a stack suffix matching $e$.
    }

    \uncover<5->{
    In our example:
    $[exp\,] = num\,\big|\,exp + [exp\,]$
    }
  }
  \end{minipage}
\end{frame}

\begin{frame}{Also...}
  The implementation also have extra features useful in practice:
  \begin{itemize}
    \item captures (to refer to semantic values and locations on the stack)
    \item ``greedy'' quantifiers (for ambiguous matches, should we keep the longest or the shortest result?)
    \item partial rules (can resume matching, like ``guarded'' patterns)
    \item total order on clauses to resolve ambiguities
    \item \ldots
  \end{itemize}

\end{frame}

\section{Practical work}

\begin{frame}{Adding an error message to OCaml}

  To add a new error message:
  \begin{itemize}
    \item start from an example, usually an issue reported by a user
    \item throw this example to an ``interpreter'' that provides information on the LR stack and {\em handles}
    \item then its our work to generalise from this example
  \end{itemize}
  \pause
  \begin{block}{If you happen to use OCaml: help!}
    Don't hesitate to report interesting errors you stumble upon.
  \end{block}
\end{frame}

\section{Conclusion}

\begin{frame}{Conclusion}
  LRgrep is a DSL ({\em domain specific language}) to identify errors of LR parsers.

\

  The first results are encouraging: we can precisely identify a rather large class of errors for the OCaml grammar. Tools are provided to help the author work out error patterns.

\

  The approach builds upon a new application for a well-understood theory, which gives hope for rapid progress.

\end{frame}

\begin{frame}{Future work}

  The main feature that is missing is a tool to test coverage of a grammar:
  which grammatical constructs have no associated errors, which rules are unreachable.

  We identified two difficulties:
\begin{itemize}
  \item computing the {\em exact} coverage itself
  \item displaying the outcome to the user
\end{itemize}

This feature will be crucial to maintain the specfications (as a grammar evolve).

\

Application to other grammars. We started looking at other candidates (only Menhir grammars for now): Catala, EasyCrypt, ...
An external contributor is looking at Lua grammar.

\end{frame}
